gtd · target state machine · design review

One repo state in,
one prompt out.

Every invocation derives a single state from three inputs — the working tree, the steering files present, and the last commit message — runs deterministic actions, and prints the next prompt. Auto-advance states tell the agent to re-run gtd, so deterministic chains run hands-free; human gates stop and wait. This is the redesign in STATES.md — not yet implemented.

How a run works

The three-step procedure

The previous agent always leaves its output uncommitted. The next gtd run infers what just happened, commits it, and hands off.

01 — DERIVE

Read the repo

Working-tree status + steering files present + last commit message → exactly one active state.

02 — ACT

Deterministic side effects

Commit pending work, revert, edit steering files, or run the test command. No model judgement.

03 — PROMPT

Emit the next instruction

Print the prompt for that state. Auto-advance states append “run gtd again”; gates stop for a human.

Detection model

Precedence ladder — first match wins

Overlapping filesystem shapes are made deterministic by a fixed priority order. Steering files are authoritative; the last commit message disambiguates what the tree alone cannot.

0HEAD is gtd: transportTransport
1ERRORS.md presentEscalate · STOP
2FEEDBACK.md present — content / emptyFixingClose pkg
3.gtd/ present — routed by tree + HEADBuild lifecycle
4REVIEW.md present — routed by commit + treeReview lifecycle
5Boundary HEAD + pending changes · no .gtd/REVIEW/FEEDBACKNew Feature
6TODO.md presentGrilling / Grilled
7Boundary / package done HEAD + clean treeCleanIdle
“Code changes” means pending working-tree changes (tracked or untracked, respecting .gitignore) outside the steering set (TODO.md, REVIEW.md, FEEDBACK.md, ERRORS.md, .gtd/). Steering-file changes are detected separately.
Linear history — state is folded from first-parent history; gtd assumes a single writer. Distribute by sequential handoff (one active machine at a time) over rebase / fast-forward — a merge commit as HEAD breaks the counter folds, the review base, and last-commit detection. gtd: transport carries uncommitted work across machines (commit, push, reset on the far side).

Commit taxonomy

Two buckets for the last commit

The last commit message is the discriminator that resolves states the filesystem can’t separate. A non-gtd commit means “cold start”.

Boundary

Cold start — no workflow in progress. Route by tree.

Mid-phase

Identifies the exact in-progress phase. The disambiguator.

new taskgrillinggrilledplanning buildingerrorsfixing feedbackpackage done awaiting review

Steering files

Authoritative — and strictly ordered

While any steering file exists, gtd resumes that workflow regardless of the last commit. They are never garbage-collected: a stale file is resumed exactly like a live one, so abandon a workflow by rm-ing its files.

Precedence when several coexist — higher wins:
ERRORS.md› FEEDBACK.md› .gtd/› REVIEW.md› TODO.md
Legal coexistence: .gtd+TODO.md (plan + packages during build), FEEDBACK.md+.gtd (fix during build).
Illegal combinations — these never occur in normal flow, so gtd hard-errors instead of guessing:
REVIEW.md + .gtd REVIEW.md + TODO.md FEEDBACK.md + REVIEW.md FEEDBACK.md without .gtd ERRORS.md + FEEDBACK.md ERRORS.md without .gtd

// review / fix · per-package — review writes FEEDBACK (empty = approve → Close); a pending review re-reviews, never skips
Two fix loops, one Fixing state. FEEDBACK arriving committed (gtd: errors, from Testing) is the test loop; arriving uncommitted (from Agentic Review) is the review loop, consumed as gtd: feedback. Both counters walk back to the package start through any non-gtd commits, so a manual commit can’t defeat escalation. The two counters never combine — test counts gtd: errors, review counts gtd: feedback. The test count resets at each gtd: feedback (the start of a review-fix) and each ERRORS.md clear, so build-phase and review-phase failures don’t accumulate and a human resume grants another N attempts.
